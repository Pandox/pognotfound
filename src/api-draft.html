Introdução

Nós, desenvolvedores, focamos nosso esforço em construir softwares com uma alta qualidade de código. Grande volume de testes unitários, integrados e principalmente código limpo são palavras chaves de todo bom programador. Aposto que livros como Clean Code e The Pragmatic Programmer surgem em conversas dessa natureza. Mas muitos de nós não focamos em uma das partes que, pessoalmente, acredito ser uma das mais importantes. A API do seu sistema.

API

Application Programming Interface, ou simplesmente API, é uma lista de rotinas, ferramentas e protocolos para construir software. Ela é usada para os programas se comunicarem entre si. Por exemplo, o Windows possui uma API para receber dados vindos de um device (teclado, mouse) e uma outra API para o Word poder ler esses dados e exibir na tela para o usuário. O Microsoft Word não se comunica diretamente com seu teclado, ele usa o Windows através de suas APIs para poder utilizar o teclado do usuário.

Desenvolver uma API e utilizá-la gerão forças totalmente diferentes e inversas. No momento da construção, você usa o chápeu de desenvolver e gasta o maior esforço possível para melhorar a qualidade do seu sistema internamente.
Ao mesmo tempo, quando você se torna o utilizador de uma API, você quer gastar o menor esforço possível para poder utilizar aquele serviço. E se caso essa API seja mal construída, você passará por stress.

Se você já teve que utilizar a API do Facebook, você sabe do que eu estou falando (facebook wins worst API in developer survey).

Como você não pode se dar ao luxo de possuir uma base de dados de 1 bilhão de usuários, fazer uma boa API é essencial para aumentar a adoção de clientes.

Uma API é a UI de desenvolvedor. É como outra UI qualquer, de atenção a ela e seus clientes irão voltar!
