Introdução

Nós, desenvolvedores, focamos nosso esforço em construir softwares com alta qualidade de código. Grande volume de testes unitários e código limpo são palavras chaves de todo bom programador. Livros como Clean Code e The Pragmatic Programmer surgem em conversas dessa natureza. Mas muitos não focam esse esforço em suas APIs.

O que é uma API?

Application Programming Interface, ou simplesmente API, é uma lista de rotinas, ferramentas e protocolos para construir software. Ela é usada para os programas se comunicarem entre si. Por exemplo, o Windows possui uma API para receber dados vindos de um device (teclado, mouse) e uma outra API para o Word poder ler esses dados e exibir na tela para o usuário.

Ser o desenvolvedor de uma API e ser o consumidor dela geram interesses conflitantes. No momento da construção, você quer gastar o maior esforço possível para melhorar a qualidade do seu sistema internamente.
Ao mesmo tempo, quando você se torna o consumidor, você quer gastar o menor tempo possível para poder utilizar aquele serviço. E se caso a API seja mal construída, você ficará bem nervoso com ela.

Se você já teve que utilizar a API do Facebook, você sabe do que eu estou falando (facebook wins worst API in developer survey).

Como você não pode se dar ao luxo de possuir uma base de dados de 1 bilhão de usuários, fazer uma boa API é essencial para aumentar a adoção de clientes.

Uma API é a UI de desenvolvedor. Faça bom proveito da UX e seus clientes voltarão!

Por que desenvolver uma boa API?

APIs de sucesso capturam clientes. Fazem eles usarem seus serviços de forma mais rápida.

Desenvolva uma API díficil de se lidar e veja o número de ligações para a sua equipe de suporte aumentarem.

Aumento do orgulho próprio! Seja conhecido como O DONO da melhor API da empresa.

Características de uma boa API

  • Fácil de usar, mesmo sem documentação,
  • É extensível,
  • Atende os requisitos de negócio,
  • Confiável.

Veja a seguir os 5 passos de como construir uma API de sucesso.

As 5 dicas de ouro

1. A API deve respeitar o SRP

Single Responsability Principle define que cada elemento deve ser responsável por uma única funcionalidade do software. Portanto, sua API deve fazer apenas uma única coisa, e fazê-la corretamente. Nem a mais, nem a menos.

2. A API deve ser suficiente para atender o negócio

Não construa funções a mais do que você precisa no momento. Over-engineering é uma má prática e deve ser evitada. Isso pode dificultar o uso de sua API.

Não construa um canivete suíço se sua missão é apenas abrir uma garrafa.

3. Os nomes importam sim

Não abrevie variáveis, não crie trocadilhos ou brincadeiras no código. Crie métodos claros e de fácil entendimento.

Está na dúvida se seu código é fácil de entender? Post nas redes sociais ou em sites de perguntas e respostas, deixe o link aqui nos comentários que será um prazer ajudá-lo.

4. Minimize o acesso a informação.

Exponha o menos possível para seus clientes. Isso facilita a testabilidade da sua API, bem como o fácil entendimento dela (afinal, o cliente tem acesso a poucos dados). Em algumas linguagens utilizamos a keyword private ou interfaces para impedir o acesso a dados e implementações.

Menos é sempre mais!

5. Documentação

Mesmo que seus métodos sejam claros e bem escritos, uma documentação dizendo o que esperar daquele métodos, os tipos de retorno, quando ocorrem suas excessões e quais são os pontos que os desenvolvedores devem se preocupar é essencial. Isso diminui a escala de aprendizagem de sua API, facilitando o uso dela.

Só tome cuidado para manter esta documentação atualizada. Se para entender o funcionamento de uma função o cliente for obrigado a ver a implementação dela, é um sinal de má prática.
